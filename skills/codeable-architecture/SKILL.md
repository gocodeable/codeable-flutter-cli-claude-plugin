# Codeable CLI Clean Architecture

You are an expert in the Flutter Clean Architecture pattern used by projects generated with `codeable_cli`. Apply these patterns when writing, reviewing, or modifying code.

## Project Structure

```
lib/
├── app/view/                    # App entry: MultiBlocProvider, MaterialApp, Splash
├── config/                      # Flavors (dev/staging/prod), API environment, Remote Config
├── constants/                   # AppColors, AppTextStyle, AssetPaths, constants
├── core/
│   ├── api_service/             # Dio HTTP client, auth interceptor, logging
│   ├── app_preferences/         # Hive local storage
│   ├── di/                      # GetIt dependency injection
│   ├── endpoints/               # API endpoint constants
│   ├── enums/                   # Shared enums
│   ├── locale/cubit/            # Language switching cubit
│   ├── models/                  # Shared DTOs (api_response, auth, profile, home, designs)
│   ├── notifications/           # Firebase + local notifications
│   ├── permissions/             # Camera, location, photo permissions
│   └── services/                # Custom business services
├── features/
│   ├── {role}/                  # Role-based features (customer, admin, business, shared)
│   │   └── {feature_name}/      # Individual feature modules
│   └── shared/                  # Cross-cutting features
├── go_router/                   # Routes, router config, exports barrel
├── l10n/                        # Localization (arb files, generated classes)
└── utils/
    ├── extensions/              # Dart extensions
    ├── helpers/                 # DataState, RepositoryResponse, validators
    └── widgets/core_widgets/    # Reusable UI components
```

## Feature Module Structure

Every feature follows this exact layered pattern:

```
feature_name/
├── data/
│   ├── models/                  # Feature-specific DTOs (optional)
│   └── repositories/            # Repository implementations (calls ApiService)
├── domain/
│   └── repositories/            # Abstract repository interfaces
└── presentation/
    ├── cubit/
    │   ├── cubit.dart           # Single cubit per feature
    │   └── state.dart           # Single state per feature
    ├── views/                   # Screen widgets (full pages)
    └── widgets/                 # Feature-specific extracted widgets
```

## Critical Rules

1. **ONE cubit and ONE state file per feature.** Never create extra cubit/state files. Add new fields to the existing state and new methods to the existing cubit.
2. **Feature widgets go in `presentation/widgets/`** of that feature, not in views or in core widgets.
3. **Each file should have only ONE widget class** (one `build` method per file). Extract child widgets into separate files in the widgets folder. No private `_build` methods — if a section of UI is complex enough to be a method, extract it into its own widget file in `presentation/widgets/`.
4. **Screens go in `views/`**, reusable parts go in `widgets/`.
5. **ApiService is a singleton** — instantiate directly in repo impl as `final ApiService _apiService = ApiService();`. NEVER inject via constructor.
6. **Repository responses** are wrapped in `RepositoryResponse<T>`.
7. **API responses** are parsed via `ResponseModel.fromApiResponse()`.
8. **All async data in state** uses `DataState<T>` — never raw types.
9. **Business logic in cubit, not UI** — Time calculations, event payload construction, refresh orchestration, data grouping/filtering belong in cubit methods. UI only calls cubit methods and renders state.
10. **Pure utilities in helpers, not repositories** — Date math, formatting, validation utilities should be static methods in helper classes (e.g., `CalendarHelper`, `DateTimeHelper`), not repository methods. Repositories are strictly for data access (API calls, cache reads).

## Data Flow

```
UI (Screen/Widget)
  → context.read<FeatureCubit>().method()
    → Cubit emits loading state
    → Repository interface method (injected via constructor)
      → Repository impl calls ApiService
        → Dio HTTP request with auth interceptor
      → Returns RepositoryResponse<T>
    → Cubit emits loaded/failure state
  → BlocBuilder rebuilds UI
```

## DI & Registration

- Repositories registered in `core/di/` via GetIt
- Cubits registered in `app/view/app_page.dart` via `MultiBlocProvider`
- Cubit receives repository via constructor injection
- ApiService is NOT injected — it's a singleton accessed directly

## Routing

- Route paths: `AppRoutes.featureScreen` (e.g., `AppRoutes.customerHomeScreen`)
- Route names: `AppRouteNames.featureScreen`
- Routes registered in `go_router/router.dart`
- Screen imports in `go_router/exports.dart`
- Navigate: `context.push(AppRoutes.xxx)` or `context.go(AppRoutes.xxx)`

## Build Flavors

Three pre-configured flavors:
- `production` — clean app name, production API
- `staging` — `[STG]` suffix, staging API
- `development` — `[DEV]` suffix, dev API, DevicePreview enabled

## Imports

Use the barrel export for common imports:
```dart
import 'package:{project}/exports.dart';
```

Import feature cubits directly:
```dart
import 'package:{project}/features/{role}/{feature}/presentation/cubit/cubit.dart';
import 'package:{project}/features/{role}/{feature}/presentation/cubit/state.dart';
```
